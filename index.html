Для контроля изменений, git использует хеши

В другом случае, первым действием будет создание папки на локальном компьютере и выполенние команды git init

В проекте так же зачастую встречаются какие то служебные файлы, создаваемы средой разработки или еще чем то, их игнорирование настраивается через папку с названием .gitignire

Один из работающих подходов в git, это иметь 2 ветки:
1 - master
2 - dev
И как только в dev что-то заработало и прошло все тесты, это кидается в мастер. И так переодически проет добавляет в себя новые фичи

  • Создание ветки происходит как git branch branchname
  • посмотреть все вети - git branch
  • для переключения на нужную ветку git checkout branchname

после изменени/добавления новой ветки и переключения на нужную происходит стандартная процедура добавления и коммита, изменения которых будут касаться выбранной заблоговременно ветки

Для объединения веток проделывается следующее:

  • Переключиться на ветку в которую хотим влить изменения (Так как откуда и куда имеет значение)
  • Выполнить команду git merge branchname - Данно слияние будет представлять собой точку пересечения веток, которой можно и имеет смылс дать говорящее название

В случае сложных проектов, имеет смысл использовать GUI интерфейс типа gitk, в котором можно наглядно отслживать ход разработки каждой ветки, их слияний и пр.

Дополнительно git status - покажет в какой ветке нахожусь и прочую информацию о текущем состоянии проекта, изменения

git log - покажет список коммитов, которые есть

Fork - форкнуть на github, значит что чужой репозиторий, появится на моем гитхабе(аккаунте), буд-то бы он мой. и я смогу его далее склонировать и работать с ним.

git remote show origin - покажет на какой удаленный репозиторий настроен наш локальный

git diff - покажет наши изменения
